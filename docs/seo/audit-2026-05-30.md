# SEO Audit — Astraa (www.astraa.tech)

**Audit Date:** 2026-05-30  **Branch:** seo/audit  **Auditor:** Claude Code /seo audit

## 1. Executive Summary

Astraa.tech is a technically competent Next.js 16 utility toolkit with the right metadata scaffolding (canonical tags, sitemap, JSON-LD, robots.txt, OG/Twitter cards, Google + Bing verification) but it consistently underperforms on the signals search engines and AI systems weigh most heavily: passage-extractable content, identity/E-E-A-T, and accurate freshness signals. Every tool page is a ~30-word UI shell with one H1, no H2 structure, no supporting prose, no per-tool structured data, and the homepage hides every tool behind client-side rendering — Googlebot cannot follow a single static link from `/` to `/tools/password`. The single biggest risk is a structural one: the homepage description claims "15+ tools" while only 8 are live and Google has zero static internal link path to those 8, so head-term and tool-name queries currently route around the site. The single biggest quick win is creating `public/llms.txt` plus adding a static, server-rendered tool grid to `app/page.tsx` — a sub-two-hour change that simultaneously fixes the crawl dead-end, unlocks per-tool PageRank flow, and positions the site for AI citation.

## 3. Critical Findings (must fix immediately)

- **Homepage is a crawl dead-end — no static HTML links to any tool page** — Critical — `app/page.tsx` + `components/home/hero-section.tsx`. Static HTML contains only 12 anchors (logo, /contribute, /explore, social externals). Every `/tools/[slug]` URL is reachable only after JS hydration via `ToolsContext`. Googlebot does execute JS but delayed-indexes JS-discovered links and passes no PageRank through sitemap entries — every tool page is currently receiving zero internal link equity from the homepage. **Fix:** render a server-side `<section>` below the hero that maps `toolCategories.flatMap(c => c.items).filter(t => !t.comingSoon)` to `<Link>` tags with the tool name as anchor text.

- **JSON-LD serialized as a bare top-level array — likely silently ignored by Google** — Critical — `app/layout.tsx:101-143`. The three schema blocks (Organization, WebApplication, WebSite) are emitted inside a single `<script type="application/ld+json">` as `JSON.stringify([{...},{...},{...}])`. Google's structured data parser requires either a single object or an object with a `@graph` property at the top level; a bare array fails Rich Results Test and Schema Markup Validator. **Fix:** split into three separate `<script>` tags or wrap in `{ "@context": "https://schema.org", "@graph": [...] }`.

- **WebSite SearchAction targets `/explore?q={search_term_string}` but `/explore` does not read the `q` param** — Critical — `app/layout.tsx:131-142` + `components/explore/explore-client.tsx`. Google validates SearchAction targets during indexing; a non-functional URL kills Sitelinks Search Box eligibility and degrades trust in the WebSite schema. **Fix:** either implement `useSearchParams().get("q")` as initial filter state in `ExploreClient`, or remove the `potentialAction` block entirely until the feature ships.

- **Tool pages are ~30 words of crawlable text — no H2, no how-to copy, no use cases, no FAQ** — Critical — every `components/[tool]/[tool]-client.tsx`. Each page renders an H1, a one-sentence tagline, the "All processing happens locally" disclaimer, and the tool widget. There is no extractable passage for AI Overviews/Perplexity/ChatGPT Browse to cite, and Sept 2025 QRG rates a page that answers no secondary questions (what is this? when do I use it? how does it work?) as low value regardless of widget quality. **Fix:** add 400–600 words of server-rendered static content per tool — H2 "What is X?", H2 "How to use it" (numbered steps), H2 "Common use cases" (3-5 bullets), H2 "FAQ" (3-5 Q&A). Do not add FAQPage schema (deprecated for most surfaces); use plain HTML.

- **`/tools/json` is live + indexable on main but absent from sitemap** — Critical — `lib/tools.ts:99-100` flags json as `comingSoon: true`; `app/tools/json/page.tsx` exists with no `robots: noindex`. Googlebot can crawl the page via any incidental internal link but the sitemap excludes it. **Fix:** either add `robots: { index: false, follow: true }` to the page metadata until PR #90 merges and `comingSoon` is dropped, or accelerate the merge.

- **Missing Content-Security-Policy on a site that handles password generation** — Critical — `next.config.js` headers section. The site makes browser-side calls to OpenRouter, CoinGecko, FawazAhmed0, and exchangerate-api. No CSP means any XSS vector exfiltrates user data. For tools positioned around browser-only privacy, this is a Trustworthiness gap. **Fix:** add `Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; connect-src 'self' https://openrouter.ai https://api.coingecko.com https://cdn.jsdelivr.net https://api.exchangerate-api.com; img-src 'self' data: blob:; style-src 'self' 'unsafe-inline'; font-src 'self'` to `next.config.js`.

- **Homepage description claims "15+ tools" but only 8 are live** — Critical — `app/page.tsx:7`, `app/layout.tsx:32`, `app/layout.tsx:65`, `app/layout.tsx:78`. The mismatch is visible the moment a user or quality rater counts tools on `/tools`. It damages Trustworthiness scoring under Sept 2025 QRG and propagates into AI summaries. **Fix:** derive the count programmatically: `const liveCount = toolCategories.flatMap(c => c.items).filter(t => !t.comingSoon).length` and interpolate.

## 4. High-Priority Findings (within 1 week)

- **No `public/llms.txt`** — `public/`. Perplexity, ChatGPT Browse, and Claude web all check for it. Zero-cost to add. Template in §9.
- **No per-tool SoftwareApplication / BreadcrumbList JSON-LD** — all `app/tools/*/page.tsx`. The global WebApplication describes the site; nothing describes individual tools. Largest missed rich-result opportunity. Code in §9.
- **Sitemap `lastModified` uses `new Date()` for all 14 URLs** — `app/sitemap.ts:14,23,31,37,43,49,55,61`. Every deploy stamps every page as freshly modified; Google distrusts the signal. **Fix:** static per-page constants imported from each page file.
- **No `/about` page, no author bio, no Person schema** — sitewide. E-E-A-T Authority + Trust gap, especially for security-adjacent tools (password, hash). **Fix:** create `app/about/page.tsx` with Aditya Kumar bio, GitHub link, Person JSON-LD with `sameAs`.
- **Homepage H1 is "Stop Searching, Start Solving" — zero topical keywords** — `components/home/hero-section.tsx`. Below-fold static content has no tool names, categories, or use-case terms. **Fix:** rewrite H1 to "Free Browser Tools for Developers and Creators — Astraa" (keep Devanagari subtitle as secondary line).
- **Markdown page title hardcodes "Markdown Editor | astraa" bypassing template** — `app/tools/markdown/page.tsx:6`. Produces either doubled "| astraa | Astraa" or inconsistent lowercase casing. **Fix:** `title: "Markdown Editor"` only.
- **`/games` indexed in sitemap (priority 0.8) but every game is `comingSoon: true`** — `app/sitemap.ts:43-46` + `lib/games.ts`. Thin-content listing page with no live children. **Fix:** gate inclusion behind `gameEntries.length > 0`.
- **Twitter card is `summary` (small image) on all tool pages while layout uses `summary_large_image`** — every `app/tools/*/page.tsx`. The 1200×630 banner gets rendered as a 1:1 thumbnail. **Fix:** change all tool pages to `card: "summary_large_image"`.
- **Logo PNG is 400KB (`public/assets/astraa_pfp.png`)** — referenced as favicon, apple-touch, manifest, Organization.logo. Hurts LCP and PWA install. **Fix:** re-export at <60KB plus WebP twin; generate 16/32/180/192 sized variants.
- **`lastUpdated` displayed on tool pages uses `new Date()` at render time** — `app/tools/{password,hash,calculator,text}/page.tsx:37`. Always shows today's date regardless of real edit history. **Fix:** static `LAST_MODIFIED` constant per page.

## 10. Sitemap & Indexing Fix

The sitemap is generated at `app/sitemap.ts` from `lib/tools.ts` by filtering `!comingSoon`. Four developer tools have production-ready pages on open feature branches but `main` still flags them `comingSoon: true`, so they are excluded from the sitemap:

| PR | Tool | Path | Status on main |
|---|---|---|---|
| #89 | Code Snippet Generator | `/tools/snippet-generator` | comingSoon=true, no page file |
| #90 | JSON Validator | `/tools/json` | comingSoon=true, page file present but no `noindex` — indexable orphan |
| #91 | Regex Tester | `/tools/regex` | comingSoon=true, no page file |
| #92 | Base64 Encoder | `/tools/base64` | comingSoon=true, no page file |

**Recommended remediation:**

1. **Immediately** (before any merge): add `robots: { index: false, follow: true }` to `app/tools/json/page.tsx` metadata. This is the one indexable orphan today.
2. **Merge order:** #92 base64 → #89 snippet → #91 regex → #90 json. Lower-complexity tools first so any merge-conflict surface is small. Each PR should drop the `comingSoon: true` flag in `lib/tools.ts` and remove `robots: noindex` from its page.
3. **Cut a single `release(v1.1.0): merge development to main` tag** after all four land. The Vercel deploy regenerates `sitemap.xml` and the four new URLs auto-appear at priority 0.8. No manual sitemap edit required.
4. **Post-merge:** ping Google Search Console "Validate Fix" on the previously orphaned `/tools/json`, and submit `/tools/{base64,snippet-generator,regex,json}` for indexing. Update the "15+ tools" copy to reflect the new live count (12 if all four land).

Until the merge, those four pages receive zero internal links, zero sitemap inclusion, and zero crawl signal. They are effectively unindexed.
